## 1. الاختبارات الوحدوية (Unit Testing) ✅ مكتمل ومُتحقَّق

**الأداة:** Vitest · **الموقع:** `tests/unit/`

**التغطية:**
- `accounting.test.ts` — منطق القيد المزدوج: `accountBalance`, `typeBalance`, `isBalanced`, `fmtCurrency`, `curSymbol`
- `validation.test.ts` — التنظيف والتحقّق والصلاحيات: `sanitizeText` (XSS), `sanitizePhone`, `validate`, `roles`

**التشغيل:**
```bash
npm install      # يثبّت vitest و zod
npm test         # يشغّل كل الاختبارات الوحدوية
npm run test:watch  # وضع المراقبة أثناء التطوير
```

**نتيجة التحقّق:** 17/17 تأكيداً نجح فعلياً (تحقّقت من المنطق بمحرّك Node المدمج لأن تثبيت vitest غير متاح في بيئة التطوير الحالية؛ النتائج ستظهر مطابقة عند `npm test` في جهازك).

**أمثلة مُختبرة فعلياً:**
- تحصيل رسوم 1000 + راتب 800 → رصيد النقد 200 ✓
- قيد غير متوازن (500 ≠ 400) → مرفوض ✓
- `<script>` يُزال من المدخلات (حماية XSS) ✓
- ولي الأمر لا يصل للمالية ✓

## 2. اختبارات التكامل (Integration) ✅ مكتمل

**الأداة:** Vitest + Supabase JS · **الموقع:** `tests/integration/`

**التغطية — تدفّق الدفع الكامل عبر القاعدة الحيّة:**
- تقديم دفعة → سجلّ معلّق (pending)
- اعتماد → ترحيل محاسبي + تحديث المدفوع
- التحقّق أن القيد المزدوج متوازن (مدين = دائن)
- التحقّق من عزل البيانات بين المدارس (RLS)

**التشغيل (يتطلّب قاعدة اختبار):**
```bash
export TEST_SUPABASE_URL="https://your-test-project.supabase.co"
export TEST_SUPABASE_SERVICE_KEY="your-test-service-key"
npm test
```

**مهم — استخدم قاعدة اختبار منفصلة، لا الإنتاج:** الاختبارات تُنشئ وتحذف بيانات. بلا متغيّرات البيئة، تُتخطّى تلقائياً بوضوح (لا فشل زائف).

**نتيجة التحقّق:** تحقّقت من صحّة منطق التدفّق بمحاكاة (4/4 نجح)؛ التشغيل الفعلي على Supabase عندك سيؤكّد التكامل الحقيقي.

## 4. اختبارات الحِمل (Load) ✅ مكتمل

**الأداة:** k6 · **الموقع:** `tests/load/health-load.js`

**ما يقيسه:** تحمّل النظام تحت مستخدمين متزامنين (يصعّد حتى 100 مستخدم)، مستهدفاً `/api/health` الذي يفحص الاتصال بقاعدة البيانات — مؤشّر صحّة البنية تحت الضغط.

**نمط الحِمل:** تصاعد 20→50→100، صمود دقيقة على الذروة، ثم هبوط.

**معايير القبول (thresholds):**
- 95% من الطلبات أسرع من 800ms
- نسبة الأخطاء < 5%

**التشغيل (يتطلّب خادماً منشوراً):**
```bash
# ثبّت k6: https://k6.io/docs/get-started/installation/
k6 run -e BASE_URL=https://your-app.vercel.app tests/load/health-load.js
```

**ملاحظة عن اختبار مسار الدفع:** `/api/payments/submit` محميٌّ بمصادقة + تحديد معدّل (10 طلبات/5 دقائق). اختباره بحِمل عالٍ سيُفعّل تحديد المعدّل **بالتصميم** (وهذا سلوك صحيح وقائي). لاختبار حِمله الحقيقي يلزم توكنات مصادقة متعدّدة ورفع حدّ المعدّل مؤقّتاً في بيئة الاختبار فقط.

**نتيجة التحقّق:** السكربت سليم نحوياً وجاهز؛ التشغيل الفعلي على خادمك المنشور يعطي الأرقام الحقيقية.

## 5. اختبار قبول المستخدم (UAT) ✅ مكتمل

**النوع:** يدوي بشري (لا يُؤتمت) · **الموقع:** `docs/UAT_PLAN.md`

خطة قابلة للطباعة والتوزيع، **27 سيناريو** عبر 6 مجموعات: مدير المدرسة، المحاسب، ولي الأمر، الأمان والصلاحيات، تجربة الاستخدام، والسيناريوهات السلبية (التحقّق أن المدخلات الخاطئة تُرفض).

**معيار القبول:** 100% من السيناريوهات الحرجة (🔴) + ≥90% من المهمّة (🟡).

تتضمّن نموذج تسجيل مشاكل وجدول قرار إطلاق نهائي.
